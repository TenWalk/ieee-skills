# Numerical Results, Discussion, and Conclusion (IEEE Communications)

## Numerical Results — build an evidence ladder, not a simulation diary

Order subsections so each rung answers one question and the rungs build toward the main claim:

```
simulation setup (parameters table) → analysis-vs-simulation validation → performance vs benchmark
schemes → parameter sweeps (SNR / antennas / users) → design analysis ("w/o" variants) →
convergence & complexity → robustness (imperfect CSI) / limitations
```

Each subsection opens **claim-first**, then gives the data:

- `The proposed scheme achieves a higher sum rate than the benchmark schemes across the SNR range.`
  then the figure and the reading.

### Explain the phenomenon, do not restate the number

Every results paragraph should carry three layers: what was observed, what it means, and how it
supports the paper's design or analysis.

Weak:

> The proposed scheme achieves a lower BER than ZF.

Strong:

> The proposed scheme achieves a lower BER than ZF, and the two curves share the same high-SNR
> slope, indicating that the gain comes from the improved array gain rather than a change in
> diversity order.

The point of analysis is *why this curve supports the paper's claim*, not *by how many dB we won*.

### Validate analysis before comparing performance

When the paper derives a closed form, the first results figure should overlay Monte-Carlo markers
on the analytical curve and confirm the asymptotic slope (diversity order / DoF). That agreement is
the evidence the derivation is correct; only then do the performance comparisons carry weight.

### Dual-metric results (ISAC and other multi-objective designs)

When the contribution serves two competing objectives (communication and sensing, rate and
energy), report **both metrics and the tradeoff between them** — a win on one axis while the other
silently degrades is not a result. The signature figure is the Pareto boundary (e.g. rate vs CRB),
generated by optimizing one metric under a constraint on the other and sweeping the constraint.
Read it as a frontier: "the proposed design pushes the rate–CRB boundary outward relative to the
orthogonal resource split", not "the rate is higher". State the shared budget (power, aperture,
bandwidth) the two functions divide. For *designing* the dual-metric evaluation, use
`ieee-experiments`.

### Each figure/table answers one question

Before writing the analysis, label each figure: validation, performance, a parameter sweep, design
analysis, convergence, or complexity. If two answer the same question, cut one. Let figures/tables
carry numbers and prose carry reasons (see `body-revision.md`, table/prose division of labour).

### Declare the fairness boundary of comparisons

If no existing scheme exactly matches the setting, say so rather than implying perfect parity:

> Since no existing scheme fully matches the considered setup, representative implementable schemes
> are adopted under the same power budget, CSI assumption, and bandwidth.

State which schemes are conventional/heuristic, which are prior-art, which are upper/lower bounds,
and the shared resource constraint. This pre-empts the most common reviewer objection. For
*designing* the comparison, use `ieee-experiments`.

## Discussion (when separate from Results)

```
central advance → what the evidence means → relation to prior work → constraints → outlook
```

Interpretation and limitations belong here. Do not replay the Results figure by figure. State
limitations honestly and bounded — a credible limitation strengthens, not weakens, the paper.

## Conclusion

```
contribution → decisive evidence → implication → boundary / future work
```

- No new data, no new figures, no re-derived equations.
- Return to the Introduction's problem and confirm what was resolved.

Skeleton:

```
This [paper/article] investigated [problem] in [system/setting].
By [formulating/deriving/designing] [approach], we developed [scheme/algorithm].
Unlike [prior approaches], the proposed scheme [distinction].
Simulation results verified that [result, e.g. the X dB gain / the derived expression / convergence].
Future work will [bounded extension, e.g. imperfect CSI, multi-cell].
```

The conclusion closes the paper; it is not a second abstract or a summary of every experiment.

## Cross-section consistency check

After drafting, verify: the conclusion's claims appear in the contributions list; every
contribution has supporting evidence in Results; terminology, scheme names, benchmark-scheme names,
metric names and figure legends are identical across all sections.
